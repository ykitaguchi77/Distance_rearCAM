## LiDAR距離測定アプリ初期実装

### 概要 (Overview)
リアカメラ（他撮り）で前方の人に対する距離をLiDARで測定するiOSアプリの初期実装を完了。
Visionフレームワークで顔検出し、ARKit LiDARの深度データから距離を取得する構成。

### 変更元ファイル (Source file)
- `ContentView.swift` (既存→書き換え)
- `ARSessionManager.swift` (新規)
- `FaceOverlayView.swift` (新規)
- `CameraView.swift` (新規)
- `project.pbxproj` (Info.plistキー追加)

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| ContentView | Hello Worldテンプレート | NavigationStack + LiDARチェック + Start画面 |
| AR機能 | なし | ARWorldTrackingConfiguration + sceneDepth |
| 顔検出 | なし | VNDetectFaceRectanglesRequest (3フレームに1回) |
| 距離測定 | なし | LiDAR depthMapから顔中心の深度読取 + 5フレーム移動平均 |
| オーバーレイ | なし | Canvas描画で緑bbox + 距離テキスト |
| スクリーンショット | なし | ARView + オーバーレイ合成でフォトライブラリ保存 |
| Info.plist | カメラ権限なし | NSCameraUsageDescription + NSPhotoLibraryAddUsageDescription |

### 変更理由 (Reason for change)
プランに基づく初期実装。iPad Pro 11インチ（LiDAR搭載）で30cm〜2mの距離測定を想定。

### アーキテクチャ詳細
- **座標変換**: Vision `.right` orientation → portrait正規化座標 → SwiftUI/深度マップへの変換
  - SwiftUI: `y = (1 - bbox.maxY) * viewHeight`
  - Depth map: `col = (1 - visionY) * width`, `row = (1 - visionX) * height`
- **concurrency**: `DispatchQueue`ベースのバックグラウンドVision処理、MainActor UIバインディング
- **スロットリング**: 3フレームに1回処理 (~10fps)、`isProcessing`フラグで重複防止

### ハマった点と解決策 (Troubleshooting)
- **座標系変換**: Vision (origin=左下, Y上) / Depth map (landscape, origin=左上) / SwiftUI (portrait, origin=左上) の3座標系の対応が複雑
  - 解決: `.right` orientationでVisionをportrait空間に統一し、depth mapへは逆回転で変換
- **SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor**: 全型がデフォルトMainActorのため、バックグラウンド処理には`nonisolated`が必要
  - 解決: `nonisolated static func`でdepth読取を分離、`DispatchQueue`で明示的にスレッド制御

## 顔認識安定化改善

### 概要 (Overview)
実機テストで距離測定は正常動作を確認。顔認識が不安定（bboxちらつき・ジッター）だったため、
フレーム間トラッキング + EMAスムージング + 消失猶予の仕組みを追加。

### 変更元ファイル (Source file)
- `ARSessionManager.swift` (大幅改修)

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| フレーム間追跡 | なし（毎回新規UUID） | TrackedFace構造体でID永続化 + 中心距離マッチング |
| bboxスムージング | なし | EMA (alpha=0.35) で位置・サイズを平滑化 |
| 距離スムージング | グローバル1バッファ | 顔ごとに個別の5フレーム移動平均バッファ |
| 消失猶予 | 0 (即消え) | maxMissCount=8 (検出途切れても8回分保持) |
| 処理頻度 | 3フレームに1回 (~10fps) | 2フレームに1回 (~15fps) |
| FaceData.id | 毎回新規UUID | TrackedFaceから引き継ぎ（安定ID） |

### 変更理由 (Reason for change)
- Visionの顔検出は毎フレーム微妙にbboxがずれる → EMAで平滑化
- 検出が1-2フレーム途切れるだけでbboxが消える → missCountで猶予
- 毎回新UUIDだとSwiftUIが毎フレームView再生成 → 安定IDで効率化

### 安定化パラメータ
| パラメータ | 値 | 説明 |
|-----------|-----|------|
| smoothingAlpha | 0.35 | EMA係数。小=滑らか・遅延大、大=追従速い・ジッター残る |
| maxMissCount | 8 | 検出途切れ後の保持フレーム数 |
| matchThreshold | 0.15 | 同一顔判定の中心距離閾値（正規化座標） |
| 処理間隔 | 2フレーム | ~15fps、スムージングとのバランス |

## 人物検出への切替 + VNTrackObjectRequestハイブリッド

### 概要 (Overview)
顔検出(VNDetectFaceRectanglesRequest)の不安定さを解消するため、人物検出(VNDetectHumanRectanglesRequest)に切替。
さらにVNTrackObjectRequestを組み合わせたハイブリッド方式で、フレーム間の追跡精度を向上。

### 変更元ファイル (Source file)
- `ARSessionManager.swift` (検出方式全面改修)

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| 検出リクエスト | VNDetectFaceRectanglesRequest | VNDetectHumanRectanglesRequest |
| フレーム間追跡 | EMAスムージングのみ | VNTrackObjectRequest + EMAスムージング |
| 検出頻度 | 毎フレーム検出 | 10フレームに1回検出、間はトラッキング |
| 深度サンプル位置 | 顔中心 | 上半身 (bbox.minY + height*0.75) |
| トラッキング信頼度 | なし | confidence > 0.15 でフィルタ |

### 変更理由 (Reason for change)
- 顔検出は距離や角度で不安定 → 人物全体の検出の方がロバスト
- 毎フレーム検出は重い → 検出+トラッキングのハイブリッドで効率化

## bbox座標系修正 (180°フリップ)

### 概要 (Overview)
実機テストでbboxのXY座標が反転していた問題を修正。
`displayTransform`方式は2回試行して失敗したため、手動マッピング+180°フリップで解決。

### 変更元ファイル (Source file)
- `ARSessionManager.swift` (座標変換ロジック修正)
- `FaceOverlayView.swift` (座標変換を削除、displayRect直接描画)

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| bbox X座標 | `bbox.minX * viewWidth` | `(1 - bbox.maxX) * viewWidth` |
| bbox Y座標 | `(1 - bbox.maxY) * viewHeight` | `bbox.minY * viewHeight` |
| Vision orientation | デバイス向き依存で動的切替 | 同左 |
| depth map変換 | orientation依存のswitch分岐 | 同左 |
| matchThreshold | 100pt (viewport座標) | 同左 |

### 変更理由 (Reason for change)
- Vision bbox出力がデバイス向き回転後のportrait空間だが、X・Y両軸が画面表示と逆
- `displayTransform`は2回試行したが、(1)orientation除去で検出失敗、(2)逆回転変換でbbox消失
- 手動マッピングで180°フリップ（両軸反転）が最もシンプルかつ確実

### ハマった点と解決策 (Troubleshooting)
- **displayTransform試行1**: Vision orientationを除去 → 顔が90°回転状態で検出失敗
- **displayTransform試行2**: orientation保持 + bbox→native座標逆変換 → bboxが画面外に飛んで消失
- **解決**: 手動マッピングに戻し、`x: (1-bbox.maxX)*w, y: bbox.minY*h` で180°フリップ
- **教訓**: `displayTransform`はARKitのcapturedImage用の変換行列であり、Vision orientationで回転済みのbboxには直接適用できない

### 最終的な座標変換まとめ
```
Vision bbox (0-1, orientation回転済み)
  → displayRect: x=(1-maxX)*viewW, y=minY*viewH (180°フリップ)
  → depth map: orientation別switchで逆回転してnative座標に戻す
```

## YOLO眼瞼検出モデル統合

### 概要 (Overview)
人物検出（VNDetectHumanRectanglesRequest）からカスタムYOLO11nモデルによる眼瞼検出（Right_eye / Left_eye）に置換。
PyTorch .ptモデルをCoreMLに変換し、VNCoreMLRequestで推論。

### 変更元ファイル (Source file)
- `ARSessionManager.swift` (コア変更: モデル読込・検出置換・パイプライン更新)
- `FaceOverlayView.swift` (UI: 色分け・ラベル表示)
- `CameraView.swift` (UI: テキスト更新・エラー表示)
- `ContentView.swift` (テキスト更新)
- `models/EyelidDetector.mlpackage` (新規: CoreML変換済みモデル)

### Before / After 比較
| 項目 | Before | After |
|------|--------|-------|
| 検出方式 | VNDetectHumanRectanglesRequest | VNCoreMLRequest (YOLO11n) |
| 検出対象 | 人物全体 | 眼瞼 (Right_eye / Left_eye) |
| モデル | Appleビルトイン | カスタムCoreML (5.1MB) |
| bbox色 | 全て緑 | Right_eye=cyan, Left_eye=yellow |
| ラベル表示 | 距離のみ | R/L + 距離 |
| detectionInterval | 10 | 5 (小さい対象→頻繁な検出) |
| matchThreshold | 100pt | 50pt (小さいbbox向け) |
| maxMissCount | 10 | 5 (まばたき対応) |
| depth上限 | 10m | 5m (眼瞼検出有効距離) |
| 深度サンプル位置 | bbox.minY + height*0.75 | bbox.midY (眼瞼中心) |
| データモデル | FaceData(id, displayRect, distance) | + label, confidence |
| モデル状態表示 | なし | modelLoaded + 未読込警告 |

### 変更理由 (Reason for change)
- 研究用途: 眼瞼までの距離を正確に測定するために、汎用人物検出から眼瞼特化モデルへ切替
- YOLO11nは軽量(5.3MB)でリアルタイム推論に適している

### 技術的ポイント
- **モデル変換**: `ultralytics` 8.4.10 + `coremltools` 9.0 で `.pt` → `.mlpackage`（nms=True付き）
- **座標系**: VNCoreMLRequestのboundingBoxはVision正規化座標と同じ空間 → 既存の180°フリップ変換がそのまま動作
- **ファイル名**: 元の `295+cerebhq1-20000_yolo11n_revert.pt` の `+` がXcodeで問題になるため `EyelidDetector.mlpackage` にリネーム
- **自動同期**: プロジェクトが `PBXFileSystemSynchronizedRootGroup` を使用 → mlpackageは自動認識
- **ラベル伝播**: トラッキングフレームではlabel=nilのため、検出フレームで取得したlabelをTrackedFaceに保持して引き継ぐ

### パラメータ調整の根拠
| パラメータ | 旧値 | 新値 | 根拠 |
|-----------|------|------|------|
| detectionInterval | 10 | 1 (毎フレーム) | VNTrackObjectRequest撤去→YOLO毎フレーム推論 |
| matchThreshold | 100pt | 50pt | 眼瞼bboxは人物bboxより小さい→マッチング半径も縮小 |
| maxMissCount | 10 | 2 | 補完なしのため短い保持期間で十分 |
| depth上限 | 10m | 5m | 眼瞼検出の実用的な有効距離 |
| smoothingAlpha | 0.3 | 0.4 | bbox位置のみEMA（距離は生値） |

### 追加改修: パイプライン簡素化
- VNTrackObjectRequest撤去 → 毎フレームYOLO推論のみ（Neural Engineで十分高速）
- 距離値: 移動平均 → 生値（補完なし）
- bbox位置: EMA (alpha=0.4) で視覚的スムージングのみ残す
- Metal GPU競合回避: `MLModelConfiguration.computeUnits = .cpuAndNeuralEngine`

### 追加改修: Portrait対応
- bbox→画面座標の変換式をデバイス向き別に分岐
  - Portrait: `x=minX*w, y=(1-maxY)*h`
  - Landscape: `x=(1-maxX)*w, y=minY*h`
- カメラセンサーは常にlandscape-rightで撮影するため、Visionのorientation回転後のbbox空間が向きで異なる
